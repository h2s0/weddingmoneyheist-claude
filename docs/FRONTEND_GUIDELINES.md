# 프론트엔드 React 개발 지침서

이 문서는 이 프로젝트의 React 프론트엔드를 일관되게 구현하고 검토하기 위한 실무 기준이다.
`AGENTS.md`의 프로젝트 원칙을 구체화하며, 두 문서가 충돌하면 `AGENTS.md`를 우선한다.

## 1. 적용 범위와 목표

다음 작업에 이 지침을 적용한다.

- React 컴포넌트와 화면 구현
- 위젯, 레이아웃, 사용자 인터랙션
- TanStack Query 기반 데이터 접근
- Zustand 기반 UI 상태 관리
- Tailwind CSS와 CSS Variables 기반 스타일링
- 접근성, 반응형, 성능, 프론트엔드 테스트

모든 변경은 다음 결과를 목표로 한다.

- 사용자가 보는 동작과 디자인이 명확하다.
- 타입과 모듈 경계만으로 코드의 책임을 이해할 수 있다.
- 상태의 소유권과 데이터 흐름이 한 방향이다.
- 데스크탑, 태블릿, 모바일과 라이트/다크 테마에서 정상 동작한다.
- 변경 범위에 맞는 검증 결과가 있다.

## 2. 현재 기술 기준

- React 18과 함수 컴포넌트를 사용한다.
- TypeScript `strict` 모드를 유지한다.
- Vite를 빌드 도구로 사용한다.
- 패키지 설치와 스크립트 실행은 pnpm만 사용한다.
- 서버 상태는 TanStack Query가 소유한다.
- 클라이언트 UI 상태는 Zustand가 소유한다.
- 대시보드 배치는 `react-grid-layout`이 담당한다.
- 차트는 Recharts를 사용한다.
- 스타일은 Tailwind CSS와 `src/styles/tokens.css`의 디자인 토큰을 사용한다.

새 라이브러리는 기존 도구로 해결할 수 없는 요구가 명확할 때만 도입한다.

## 3. 디렉터리와 책임

| 경로 | 책임 |
| --- | --- |
| `src/components/common` | 도메인에 의존하지 않는 공통 UI |
| `src/components/layout` | 페이지 프레임과 전역 레이아웃 UI |
| `src/components/dashboard` | 위젯 배치, 등록, 대시보드 조합 |
| `src/components/widgets` | 독립적으로 렌더링되는 도메인 위젯 |
| `src/api` | Query hook, query key, 데이터 접근 경계 |
| `src/stores` | 테마, 레이아웃 등 클라이언트 UI 상태 |
| `src/lib` | 포맷팅, 계산, 순수 유틸리티 |
| `src/mocks` | 개발용 서버 응답 대체 데이터 |
| `src/types` | 공유 도메인 타입과 UI 계약 |
| `src/styles` | 전역 스타일과 디자인 토큰 |

컴포넌트에서 데이터 요청, 비즈니스 계산, 전역 상태 변경, 레이아웃 정책을 한꺼번에 처리하지 않는다.
새 파일은 가장 구체적인 책임을 가진 디렉터리에 둔다.

컴포넌트는 `ComponentName/index.tsx` 폴더 구조를 기본으로 한다.

- `index.tsx`는 props와 조회 결과를 UI로 조합하는 렌더링 책임을 가진다.
- `constants.ts`는 여러 렌더에서 공유하는 정적 메시지, 라벨, 스타일 맵, 설정값이 있을 때만 만든다.
- `index.hooks.ts`는 지역 상태, effect, 이벤트 조합, 비즈니스 파생 계산이 있을 때만 만든다.
- 단순 API hook 호출이나 한 번만 쓰는 문자열을 옮기기 위해 빈 껍데기 파일을 만들지 않는다.
- 독립적으로 읽을 가치가 있는 하위 UI는 같은 컴포넌트 폴더에 별도 `.tsx` 파일로 둔다.

## 4. 컴포넌트 작성

### 기본 원칙

- 함수 컴포넌트와 named export를 사용한다.
- props 타입은 컴포넌트 가까이에 `interface`로 선언한다.
- `React.FC` 대신 props를 함수 매개변수에 직접 지정한다.
- 컴포넌트는 한 가지 사용자 관점의 책임만 가진다.
- 300줄을 넘기기 전에 상태 소유권과 UI 경계를 먼저 재검토한다.
- 조합으로 해결할 수 있는 UI를 거대한 조건부 컴포넌트로 만들지 않는다.
- 단순한 값을 전달하기 위해 전역 상태나 Context를 추가하지 않는다.

```tsx
interface PriceBadgeProps {
  value: number
  label: string
}

export function PriceBadge({ value, label }: PriceBadgeProps) {
  return (
    <span aria-label={`${label} ${value}원`}>
      {formatCurrency(value)}
    </span>
  )
}
```

### 렌더링 규칙

- 렌더 함수는 같은 props와 상태에 대해 같은 UI를 반환해야 한다.
- 파생 값은 가능하면 렌더 중 계산하고 별도 state로 복제하지 않는다.
- 목록 key는 데이터의 안정적인 ID를 사용한다. 배열 index를 영속 데이터 key로 사용하지 않는다.
- 조건이 복잡하면 의미 있는 변수나 작은 컴포넌트로 분리한다.
- JSX 안에 도메인 계산이나 긴 데이터 변환을 넣지 않는다.
- 사용자에게 보이지 않는 장식 요소에는 적절히 `aria-hidden`을 적용한다.

## 5. TypeScript 규칙

- `any`, `@ts-ignore`, 무근거한 type assertion을 사용하지 않는다.
- 외부 입력과 nullable 값은 경계에서 좁힌다.
- 도메인 타입은 `src/types`, 컴포넌트 전용 props는 해당 파일에 둔다.
- 같은 의미의 타입을 여러 파일에 중복 정의하지 않는다.
- 문자열 집합은 가능한 경우 union 타입이나 `as const` 원천 데이터에서 파생한다.
- 함수 반환 타입은 공용 API나 복잡한 경계에서 명시하고, 단순한 내부 함수는 추론을 활용한다.
- `unknown`을 사용한 뒤 검사하는 방식을 타입 우회보다 우선한다.
- 타입만 가져올 때는 `import type`을 사용한다.

서버 응답 타입과 화면 표시 모델이 달라지면 API 경계에서 명시적으로 변환한다. 컴포넌트가 원시 응답 구조를 해석하게 하지 않는다.

## 6. 상태와 데이터 흐름

상태는 가장 좁고 명확한 소유자에게 둔다.

| 상태 종류 | 소유 위치 |
| --- | --- |
| 입력값, 열림/닫힘 등 지역 UI | 컴포넌트 `useState` |
| 테마, 위젯 표시, 저장 레이아웃 | Zustand |
| 포트폴리오, 보유 종목, 거래, 차트 | TanStack Query |
| 포맷된 금액, 합계, 필터 결과 | 렌더 계산 또는 `src/lib` 순수 함수 |
| URL로 공유해야 하는 필터 | 라우터/URL 상태 도입 시 URL |

### Zustand

- 서버 응답과 비동기 캐시를 저장하지 않는다.
- action을 통해서만 상태를 변경한다.
- 새 코드에서는 필요한 조각만 selector로 구독해 불필요한 렌더를 줄인다.
- persist key와 저장 데이터 구조 변경은 마이그레이션 또는 버전 변경을 함께 검토한다.
- store에 DOM 노드, 컴포넌트, Promise 같은 비직렬 값을 저장하지 않는다.

### TanStack Query

- 컴포넌트가 직접 `fetch`하지 않고 `src/api`의 query hook을 사용한다.
- query key는 도메인과 파라미터를 안정적으로 표현한다.
- query function은 요청과 응답 변환만 담당하고 UI를 알지 못한다.
- loading, error, empty, success 상태를 서로 구분한다.
- 서버 데이터를 로컬 state에 복사하지 않는다.
- mutation을 추가할 때 성공 후 무효화 또는 캐시 갱신 전략을 명시한다.

## 7. Hook과 Effect

- Hook은 컴포넌트 최상위 또는 custom hook 최상위에서만 호출한다.
- `useEffect`는 DOM, 브라우저 API, 구독처럼 React 외부 시스템과 동기화할 때 사용한다.
- props나 state로 계산 가능한 값을 effect에서 다시 state로 만들지 않는다.
- effect가 등록한 listener, timer, subscription은 cleanup에서 해제한다.
- dependency 경고를 억제하지 말고 의존 관계를 단순화한다.
- custom hook 이름은 `use`로 시작하고 데이터 또는 동작의 소유권을 명확히 표현한다.

`useMemo`와 `useCallback`은 기본값이 아니다. 비용이 측정되었거나 참조 안정성이 실제 계약인 경우에만 사용하고, 그 이유가 코드에서 드러나게 한다.

## 8. 이벤트와 비동기 처리

- 이벤트 핸들러는 `handleSubmit`, `handleThemeToggle`처럼 의도를 드러낸다.
- props로 전달하는 이벤트는 `onSubmit`, `onThemeToggle`처럼 이름 짓는다.
- 버튼의 기본 동작이 의도와 다르면 `type="button"`을 명시한다.
- 사용자의 연속 입력으로 경합이 생길 수 있는 비동기 작업은 최신 요청 기준을 보장한다.
- 실패를 조용히 삼키지 않는다. 사용자가 복구할 수 있는 오류 UI를 제공한다.
- 금액 변경, 삭제 같은 위험 작업은 결과와 복구 가능성을 기준으로 확인 절차를 설계한다.

## 9. 스타일과 디자인 시스템

- `legacy/index.html`의 시각적 결과를 디자인 기준으로 삼는다.
- 색상, 그림자, radius는 `src/styles/tokens.css`의 CSS 변수를 사용한다.
- Tailwind는 배치, 간격, 타이포그래피, 반응형 표현에 사용한다.
- 색상 hex, rgb 값을 컴포넌트에 직접 작성하지 않는다.
- 임의 값 문법은 기존 디자인의 정확한 수치를 재현할 때만 제한적으로 사용한다.
- 같은 스타일 조합이 반복되면 기존 공통 컴포넌트나 CSS class를 먼저 확장한다.
- 라이트와 다크 테마 모두에서 대비, 경계선, 차트 색상을 확인한다.
- hover만으로 의미나 기능을 전달하지 않는다.

디자인 변경 요청이 아닌 작업에서 색, 간격, 카드 형태를 임의로 개선하지 않는다.

## 10. 반응형과 대시보드 위젯

- 데스크탑, 태블릿, 모바일을 모두 지원한다.
- 콘텐츠를 숨겨서 overflow 문제를 덮지 말고 정보 우선순위와 스크롤 경계를 정한다.
- 터치 대상은 충분한 크기와 간격을 갖게 한다.
- 브레이크포인트별로 텍스트 잘림, 차트 크기, 표 스크롤을 확인한다.
- 위젯 ID는 저장 레이아웃 계약이므로 한번 배포한 값을 임의로 변경하지 않는다.
- 위젯은 grid 위치를 알지 않고 주어진 영역 안에서 독립적으로 렌더링한다.
- 드래그는 지정된 handle에서만 시작되게 한다.
- 새 위젯은 `.codex/skills/add-widget/SKILL.md` 절차를 따른다.

모든 데이터 위젯은 다음 상태를 제공한다.

- loading: 레이아웃 이동을 최소화하는 skeleton 또는 진행 상태
- error: 문제 설명과 가능한 복구 동작
- empty: 오류와 구분되는 빈 결과 설명
- success: 정상 데이터와 핵심 정보

## 11. 접근성

- 상호작용에는 `button`, `a`, `input` 등 올바른 HTML 요소를 사용한다.
- 클릭 가능한 `div`를 만들지 않는다.
- 모든 기능은 키보드로 접근하고 실행할 수 있어야 한다.
- 포커스 표시를 제거하지 않는다.
- 아이콘 단독 버튼에는 목적을 설명하는 accessible name을 제공한다.
- form control은 label과 연결한다.
- 상태 변화가 시각적으로만 전달되지 않게 텍스트나 의미를 함께 제공한다.
- 모달과 패널은 포커스 이동, Escape 닫기, 배경 접근 제한을 함께 설계한다.
- 색상만으로 수익/손실 또는 성공/실패를 구분하지 않는다.

## 12. 성능

- 먼저 올바른 데이터 흐름과 렌더 경계를 만든 뒤 측정 가능한 문제를 최적화한다.
- 큰 차트 계산과 정렬/집계는 순수 함수로 분리하고 필요할 때 memoize한다.
- 전역 store 전체 구독보다 필요한 selector 구독을 우선한다.
- render 중 매번 생성되는 객체가 하위 memo 계약을 깨는지 확인한다.
- 큰 기능이나 차트의 초기 로딩 비용이 실제 문제일 때 lazy loading을 검토한다.
- 성능을 이유로 읽기 어려운 캐시와 수동 동기화를 선제적으로 추가하지 않는다.

## 13. 테스트와 검증

현재 저장소에는 unit test script가 구성되어 있지 않다. 테스트 인프라가 추가되기 전에는 다음 검증을 최소 기준으로 삼는다.

```bash
pnpm run lint
pnpm run typecheck
pnpm run build
```

UI 변경은 추가로 다음을 직접 확인한다.

- 변경된 사용자 흐름
- 데스크탑, 태블릿, 모바일
- 라이트, 다크 테마
- loading, error, empty, success 상태
- 키보드와 포커스 동작
- 브라우저 console 오류

테스트 인프라를 도입할 때는 다음 우선순위를 따른다.

1. 순수 비즈니스 로직과 포맷팅 함수의 unit test
2. 상태 전환이 중요한 컴포넌트 test
3. 대시보드 핵심 사용자 흐름의 Playwright E2E test

버그 수정에는 가능하면 실패를 재현하는 회귀 테스트를 먼저 추가한다. 테스트를 추가할 수 없다면 검증하지 못한 범위를 작업 보고에 명시한다.

## 14. 코드 리뷰 체크리스트

- 변경 이유와 성공 기준이 명확한가?
- 상태가 올바른 계층에 있는가?
- 컴포넌트가 직접 서버 데이터를 요청하거나 해석하지 않는가?
- 타입 우회와 중복 타입이 없는가?
- effect가 외부 동기화에만 사용되는가?
- loading, error, empty 상태가 구분되는가?
- 기존 디자인과 두 테마를 유지하는가?
- 모바일, overflow, 키보드 접근을 확인했는가?
- 불필요한 추상화나 새 dependency가 없는가?
- lint, typecheck, build와 필요한 동작 검증을 통과했는가?

## 15. 완료 정의

프론트엔드 작업은 구현만 끝난 상태를 완료로 보지 않는다. 다음 조건을 모두 만족해야 한다.

- 요청된 사용자 동작이 구현되었다.
- 코드 책임과 상태 소유권이 이 지침에 맞다.
- 기존 시각 디자인과 반응형 동작이 유지된다.
- 접근성과 오류 상태를 확인했다.
